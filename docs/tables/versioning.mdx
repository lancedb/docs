## Tag-Based Versioning

Numeric table versions like `v3` or `v17` are precise but hard to remember. Tags
let you attach human-readable labels (e.g., `"prod"`, `"baseline"`,
`"q3-evaluation"`) to specific versions and check those out by name. They are
conceptually similar to git tags, and unlike numeric versions, **tagged versions
are preserved when old versions are pruned** (see the cleanup note at the bottom
of this page).

The tags API supports the standard CRUD operations — create, list, update, delete —
plus checking out by tag name.

Deleting a tag only removes the label, not the version it points to. After
deletion, the underlying table version becomes eligible for cleanup again.
</Note>

## Branches

Branches are isolated, writable lines of history forked from another branch (or
a specific version). Writes to a branch do **not** affect `main`, which makes
branches ideal for:

- Running experiments (new indexes, schema changes, reprocessing) without
  disturbing production reads on `main`.
- Backfills or migrations that you want to review before promoting.
- Sharing a frozen point-in-time fork with a collaborator while continuing to
  write to `main`.

Unlike tags, which are read-only labels on existing versions, a branch has its
own writable history. Creating or checking out a branch returns a new table
handle whose reads and writes are scoped to that branch.

<Note>
Branches are currently supported on local and namespace-backed tables in
LanceDB OSS. Branch support for LanceDB Enterprise (remote) tables is not yet
available.
</Note>

### Create and Write to a Branch

Create a branch from `main` (the default source) and write to it. The base
table continues to point at `main` and is unaffected by writes on the branch.

<CodeGroup>
```python Python icon="python"
# Create a branch from main and write to it — main is untouched.
branch = table.branches.create("exp")
branch.add([{"vector": [1.1, 2.2], "item": "experiment", "price": 99.0}])

# Fork from a specific source branch or pinned version.
branch_from_v3 = table.branches.create("hotfix", from_ref="main", from_version=3)
```

```typescript TypeScript icon="square-js"
const branches = await table.branches();

// Create a branch from main and write to it — main is untouched.
const branch = await branches.create("exp");
await branch.add([{ vector: [1.1, 2.2], item: "experiment", price: 99.0 }]);

// Fork from a specific source branch or pinned version.
const hotfix = await branches.create("hotfix", "main", 3);
```

```rust Rust icon="rust"
use lancedb::table::Ref;

// Create a branch from main's latest version and write to it.
let branch = table.create_branch("exp", Ref::Version(None, None)).await?;
branch.add(batches).execute().await?;
```
</CodeGroup>

### Reopen a Branch

You can reopen an existing branch either from a table handle or directly from
the database connection when opening the table.

<CodeGroup>
```python Python icon="python"
# Check out from an existing table handle.
branch = table.branches.checkout("exp")

# Or open the table directly into the branch.
branch = db.open_table("items", branch="exp")
```

```typescript TypeScript icon="square-js"
// Check out from an existing table handle.
const branches = await table.branches();
const checkedOut = await branches.checkout("exp");

// Or open the table directly into the branch.
const opened = await db.openTable("items", undefined, { branch: "exp" });
```

```rust Rust icon="rust"
// Check out from an existing table handle.
let branch = table.checkout_branch("exp").await?;

// Or open the table directly into the branch.
let branch = db.open_table("items").branch("exp").execute().await?;
```
</CodeGroup>

### List and Delete Branches

`list` returns a mapping of branch names to branch metadata. `delete` removes
a branch and its branch-local history; data on `main` is not affected.

<CodeGroup>
```python Python icon="python"
all_branches = table.branches.list()
print(all_branches.keys())

table.branches.delete("exp")
```

```typescript TypeScript icon="square-js"
const branches = await table.branches();
const all = await branches.list();
console.log(Object.keys(all));

await branches.delete("exp");
```

```rust Rust icon="rust"
let all_branches = table.list_branches().await?;
table.delete_branch("exp").await?;
```
</CodeGroup>

<Note>
**Branches vs. tags vs. checkout**

- Use a **tag** to bookmark an existing version with a human-readable name. Tags
  are read-only and protect that version from cleanup.
- Use **`checkout(version)`** to read from a historical version of `main`
  without forking. The handle is read-only until you `restore`.
- Use a **branch** when you need to *write* on top of a point in history
  without touching `main`.
</Note>

### Version History and Operations

On a fresh table, the snippets in this guide produce this version sequence:

1. `v1`: create table (`create_table` / `createTable` / `create_table`)
2. `v2`: update rows (`update`)
3. `v3`: add rows (`add`)
4. `v4`: restore snapshot (`restore`) from `version_after_mod`/`versionAfterMod`
5. `v5`: delete rows (`delete`)

Read-only and checkout operations shown here (`list_versions`/`listVersions`, `version`, `checkout`, `checkout_latest`/`checkoutLatest`) do not create new versions.

The version metadata fields can differ by backend. Direct table-backed version listing exposes a
timestamp, while namespace-backed listing may expose fields such as `manifest_path`,
`manifest_size`, `e_tag`, and `timestamp_millis`. In deployments that use managed versioning,
prefer the table version APIs exposed by LanceDB Enterprise or the namespace service instead of
mixing in lower-level Lance file operations.

<Note>
**System Operations**

System operations like `optimize()`, index updates, and table compaction also increment table version numbers.
In LanceDB OSS and Enterprise, `optimize()` can prune older versions based on its retention setting (`cleanup_older_than`, 7 days by default),
which is when old-version files are removed and disk space is reclaimed.

**Tagged versions are exempt from cleanup.** A version with a tag pointing at it is
retained regardless of age, and its files are not removed by `optimize()`. To make
a tagged version eligible for pruning, [delete the tag](#tag-based-versioning) first.
</Note>
